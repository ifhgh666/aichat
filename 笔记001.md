# 笔记001

## 1. 这个项目的整体流程

当前项目的请求链路是：

```text
前端 index.html
-> 调用自己的后端接口 /api/chat
-> 后端 app.js 调用 DeepSeek API
-> DeepSeek 返回结果
-> 后端把结果返回给前端
-> 前端把消息显示出来
```

重点：

- 前端不直接拿 `DEEPSEEK_API_KEY` 去请求 DeepSeek。
- 前端先请求自己的后端接口 `/api/chat`。
- 后端负责保护密钥、组织参数、调用 DeepSeek、处理错误。

## 2. 为什么不能把 API Key 放前端

原因：

- 前端代码会暴露给用户。
- 如果把 key 写在前端，别人打开网页就能看到。
- 别人拿到 key 后可以消耗自己的额度。

正确做法：

- 把 key 放在 `.env`。
- 后端通过 `process.env.DEEPSEEK_API_KEY` 读取。
- 前端永远不要直接接触真实 key。

## 3. `.env` 和 `.gitignore` 是做什么的

### `.env`

`.env` 用来放本地私密配置，比如：

```env
DEEPSEEK_API_KEY=你的key
DEEPSEEK_MODEL=deepseek-chat
PORT=3000
```

代码里通过：

```js
process.env.DEEPSEEK_API_KEY
```

来读取。

### `.gitignore`

`.gitignore` 是告诉 Git 哪些文件不要提交到仓库。

当前项目里最重要的是：

```gitignore
.env
```

这样做是为了避免把真实 API Key 传到 GitHub。

### `.env.example`

`.env.example` 是模板，可以提交到仓库，告诉别人需要哪些环境变量，但不能放真实 key。

## 4. 为什么之前会乱码

前端页面出现乱码，是因为文件内容和声明编码不一致。

现在已经改成：

- 文件本身是 UTF-8
- 页面里有 `<meta charset="UTF-8" />`

如果浏览器正常但编辑器里仍然乱码，通常是编辑器打开编码不对，要在 VS Code 里切换成 `UTF-8`。

原因有几个：

- 项目里真实入口在 `1/app.js`，不是原来的 `index.js`。

- 增加了根目录 `app.js`

## 6. 为什么前端先调后端，而不是直接调 DeepSeek

这是正常项目最常见的做法。

流程：

```text
前端 -> 自己的后端 -> DeepSeek
```

这么做的原因：

- 保护 API Key
- 方便统一错误处理
- 方便以后换模型
- 方便加日志、限流、业务逻辑
- 方便接知识库、数据库、天气、订单接口

## 7. `JSON` 是什么

JSON 是一种传数据的文本格式。

例子：

```json
{
  "message": "你好"
}
```

特点：

- 用 `{}` 表示对象
- 是键值对格式
- 字符串要用双引号
- 前后端接口通信里最常见

## 8. `JSON.stringify()` 是什么

`JSON.stringify()` 的作用是：

把 JavaScript 对象转成 JSON 字符串。

例如：

```js
JSON.stringify({
  message: "你好"
})
```

会变成：

```json
{"message":"你好"}
```

这个字符串会被放进请求的 `body` 里发给服务器。

## 9. `Content-Type: application/json` 是什么

这句话是在告诉服务器：

“我发过去的数据是 JSON 格式。”

例如：

```js
headers: {
  "Content-Type": "application/json"
}
```

意思就是：

- 请求正文是 JSON
- 服务器应该按 JSON 去解析

## 10. `messages` 是什么

聊天模型通常不是直接吃一整段普通字符串，而是接收一个消息数组。

例如：

```js
messages: [
  {
    role: "system",
    content: "你是一个简洁、友好的中文助手。"
  },
  {
    role: "user",
    content: userMessage
  }
]
```

这里的角色：

- `system`：系统提示词，用来设定模型身份、语气、规则
- `user`：用户输入的内容
- `assistant`：模型之前说过的话，多轮对话时会用到

## 11. 系统提示词有什么用

系统提示词就是给模型设定人设和行为规则。

例如：

```js
{
  role: "system",
  content: "你是一只叫小白的小狗，说话可爱、忠诚、用中文回答。"
}
```

这样模型会尽量按这个身份来回答。

注意：

- 可以引导风格和角色
- 不能保证绝对无条件服从
- 遇到事实性问题，设定过强可能会让回答更不稳定

## 12. 为什么接了大模型 API 也不等于会联网

接了 DeepSeek API，不等于模型会自动上网查资料。

当前项目只是：

```text
前端 -> 后端 -> DeepSeek 模型
```

没有：

- 搜索引擎
- 天气 API
- 新闻 API
- 知识库
- 数据库查询

所以模型回答实时问题时，可能会猜错。

真正能查实时信息的产品，通常是：

```text
模型 + 搜索 / 知识库 / 外部接口 / 工具调用
```

## 13. SDK 是什么

SDK 可以简单理解成：

“别人帮你封装好的开发工具包。”

不用 SDK 时，要自己写 `fetch(...)`。

用了 SDK 后，调用会更简洁。

例如：

```js
openai.chat.completions.create(...)
```

## 14. 为什么 DeepSeek 可以用 `OpenAI` 的 SDK

不是因为在调用 OpenAI 的模型，而是因为：

- OpenAI 的接口格式很流行
- DeepSeek 兼容了这套格式

所以可以写：

```js
const openai = new OpenAI({
  baseURL: "https://api.deepseek.com",
  apiKey: process.env.DEEPSEEK_API_KEY
})
```

这里真正请求的还是 DeepSeek，因为 `baseURL` 指向的是 DeepSeek。

## 15. 流式回复是什么

流式回复的意思是：

- 模型不是等整段话全部生成完再一次性返回
- 而是一边生成，一边把内容一点一点发出来

对比：

### 非流式

- 等完整回答生成结束
- 一次性返回

### 流式

- 先返回一点
- 再返回一点
- 前端可以边接收边显示

## 16. “流”是什么

流可以理解成：

“数据不是一次性整体给你，而是一小块一小块连续传输。”

生活类比：

- 非流式像整桶水一次性给你
- 流式像水龙头持续出水

## 17. SSE 是什么

SSE 全称是：

```text
Server-Sent Events
```

可以理解成：

“服务器不断往客户端推送消息的一种格式。”

DeepSeek 开 `stream: true` 时，返回的是 SSE 风格的数据，大概像这样：

```text
data: {"choices":[{"delta":{"content":"你"}}]}

data: {"choices":[{"delta":{"content":"好"}}]}

data: [DONE]
```

含义：

- 每个 `data:` 是一条增量消息
- `[DONE]` 表示生成结束

## 18. 这次流式回复是怎么实现的

### 后端

后端做的事：

- 请求 DeepSeek 时设置 `stream: true`
- 不再用 `response.json()`
- 改成从 `response.body` 里持续读取数据
- 解析每个 SSE 数据块
- 把新文本片段用 `res.write(...)` 继续发给前端

### 前端

前端做的事：

- `fetch("/api/chat")`
- 不再用 `await res.json()`
- 改成 `res.body.getReader()`
- 每读到一段文本，就追加到当前机器人消息

所以最终效果就是：

```text
DeepSeek 边生成
-> 后端边转发
-> 前端边显示
```

## 19. 为什么之前流式会报错

原因是：

```js
stream: true
const data = await response.json()
```

这两句冲突。

原因：

- `stream: true` 返回的是流
- 流不是一次性完整 JSON
- 所以不能直接 `response.json()`

必须改成逐块读取。

## 20. 端口为什么会被占用

之前 `3000`、`3001` 被多个 Node 进程占用，通常是因为：

- 之前启动过项目
- 后台旧进程没关闭
- Code Runner 或其他启动方式留下了残留进程

后来做的处理：

- 手动关闭占用端口的 Node 进程
- 后端启动时如果端口被占用，会自动切换到下一个端口

## 23. 今天最重要的结论

- 正常项目里，大模型 key 放后端，不放前端。
- 前端先调自己的后端接口，再由后端去调 DeepSeek。
- `.env` 放私密配置，`.gitignore` 防止误传。
- 大模型 API 不等于自动联网。
- 流式回复不是一次性 JSON，而是持续读取数据流。
- `stream: true` 后，前后端都必须改成按流处理。

## 24. 产品能联网，不等于裸模型天然会联网

像 DeepSeek 网页版、豆包、Kimi 这类产品，看起来像“模型自己会上网”，但很多时候其实是：

```text
产品层 + 搜索/知识库/外部工具 + 大模型
```

也就是说：

- 裸 API 通常只是模型本身
- 网页版/App 往往是“模型 + 联网搜索 + 工具能力”的组合

所以：

- 自己接 DeepSeek API，不代表自动拥有网页版那种联网能力
- 如果想查天气、新闻、当前日期、订单信息，通常要自己在后端再接一层外部接口

## 25. 前后端的流到底是怎么拼起来的

这次项目里的流式链路是：

```text
DeepSeek
-> 你的后端
-> 你的前端
```

### 第一步：DeepSeek 给后端返回 SSE 流

DeepSeek 开 `stream: true` 后，不会一次性返回完整 JSON，而是一段一段往回发。

后端里这句：

```js
const reader = response.body.getReader()
```

就是开始持续读取 DeepSeek 的返回流。

### 第二步：后端先把“半截 SSE 数据”拼完整

后端里的：

```js
buffer += decoder.decode(value, { stream: true })
const lines = buffer.split("\n")
buffer = lines.pop() || ""
```

这里的 `buffer` 不是为了“攒很多字再传”，而是为了避免拿到半截 SSE 文本。

因为某次读到的数据可能是不完整的，例如只读到：

```text
data: {"choices":[{"delta":{"cont
```

这时还不能解析，必须等下一次读到剩下那一半再拼起来。

所以：

- `buffer` 的作用是把协议文本拼完整
- 不是故意等很多正文再一起发

### 第三步：后端一拿到正文片段就立刻转发

后端解析出本次新增的文本后，会执行：

```js
res.write(chunk)
```

这表示：

- 只要这次拿到了新的文字片段
- 就立刻往前端发

所以当前实现不是“攒够 20 个字再发”，而是：

```text
有一段就发一段
```

每次到底是一两个字、几个字、还是一个短词组，并不固定，主要取决于：

- DeepSeek 这次返回的增量大小
- 网络怎么分块
- Node 和浏览器每次读到多少

### 第四步：前端把后端发来的片段继续拼成完整回复

前端里这句：

```js
reply += decoder.decode(value, { stream: true })
```

就是把后端发来的一小段一小段文本继续拼到 `reply` 里。

所以整个过程中有两次“拼接”：

1. 后端的 `buffer`
作用：把不完整的 SSE 数据拼成完整的一行

2. 前端的 `reply`
作用：把已经提取出来的正文片段拼成完整回答

## 26. 流式回复时，一次到底传多少个字

当前实现不是固定“一次传 1 个字”或者“一次传 10 个字”。

真实情况是：

- DeepSeek 生成一点，就返回一点
- 后端解析到一点，就写一点
- 前端收到一点，就显示一点

所以一次可能是：

- 一个字
- 两三个字
- 一个词
- 一小段短句

不固定。

## 27. AI 是一个字一个字分析问题吗

不是严格按“汉字”理解和生成，而是按 `token` 来处理。

### token 可以理解成什么

可以先把 token 理解成：

“模型内部切分文本时使用的最小处理单位”

它不一定刚好等于一个汉字。

例如一句中文：

```text
今天天气怎么样
```

在模型内部不一定是按这 7 个字一一切开，它可能按别的方式分成几段。

### 模型生成回答的方式

模型生成时更像是：

1. 先看前面的上下文
2. 预测下一个最可能的 token
3. 再根据更新后的上下文继续预测下一个 token
4. 一步一步把整段话生成出来

所以：

- 它确实是逐步生成的
- 但不是简单按“一个汉字一个汉字理解世界”
- 而是按 token 在整段上下文上做预测

### 为什么流式输出能成立

因为模型本来就是逐步生成的，所以很适合：

```text
生成一点
-> 返回一点
-> 前端显示一点
```
