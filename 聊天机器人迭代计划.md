# 聊天机器人迭代计划

这个项目现在已经完成了最核心的一条链路：

```text
前端页面
-> 请求自己的 Node.js 后端接口
-> 后端携带 API Key 请求 DeepSeek
-> DeepSeek 流式返回
-> 后端转发给前端
-> 前端边接收边显示
```

后面不要急着一下子做很大，可以按版本慢慢迭代。每完成一个版本，都可以提交一次 GitHub，方便以后回看自己的成长路线。

## 版本 1：基础稳定版

目标：让当前项目更稳定、更容易调试。

要做的功能：

- 页面上显示当前连接状态，比如“后端已连接”“请求中”“请求失败”。
- 页面上显示当前模型名，比如 `deepseek-v4-pro`。
- 后端启动时打印当前端口、当前模型名，但不要打印 API Key。
- 错误提示区分清楚，比如 API Key 没配置、余额不足、模型名错误、请求超时。
- 保留现在的流式回复能力。

完成后效果：

```text
用户能清楚知道项目有没有连上后端，也能知道当前用的是哪个模型。
```

重点学习：

- Express 接口怎么返回不同状态码。
- 前端怎么根据不同错误显示不同提示。
- 什么信息可以打印，什么信息不能打印。

## 版本 2：多轮对话版

目标：让机器人记得当前页面里的上下文。

现在的问题：

```text
前端每次只发当前一句话给后端，模型不一定知道前面聊过什么。
```

要做的功能：

- 前端用数组保存聊天记录。
- 每次请求后端时，不只发送当前输入，而是发送最近几轮消息。
- 后端接收 `messages`，再转发给 DeepSeek。
- 控制上下文长度，比如只保留最近 10 轮，避免请求太大、费用太高。

完成后效果：

```text
用户问“我刚才说了什么”，机器人能根据当前聊天记录回答。
```

重点学习：

- `messages` 数组的结构。
- `system`、`user`、`assistant` 三种角色的区别。
- 为什么上下文越长，消耗 token 越多。

## 版本 3：停止生成版

目标：用户可以中断流式回复。

要做的功能：

- 前端增加“停止生成”按钮。
- 请求开始时创建 `AbortController`。
- 用户点击停止时调用 `controller.abort()`。
- 后端监听请求关闭，及时停止继续向前端写入。

完成后效果：

```text
模型正在输出长答案时，用户可以直接停止。
```

重点学习：

- `AbortController` 是什么。
- 前端怎么中断 `fetch`。
- 流式响应为什么需要处理“中途断开”。

## 版本 4：聊天记录保存版

目标：刷新页面后，聊天记录不会立刻消失。

第一阶段可以先用浏览器本地保存：

- 使用 `localStorage` 保存聊天记录。
- 页面打开时从 `localStorage` 读取历史记录。
- 增加“清空聊天记录”按钮。

第二阶段再考虑数据库：

- 使用 SQLite 或 MySQL 保存聊天记录。
- 后端增加查询历史记录接口。
- 后端增加删除历史记录接口。

完成后效果：

```text
用户刷新页面后，还能看到之前的聊天内容。
```

重点学习：

- 前端本地存储。
- 后端接口如何读写数据。
- 数据库为什么适合保存长期数据。

## 版本 5：Markdown 和代码显示版

目标：让机器人回答代码、列表、标题时更好看。

要做的功能：

- 前端支持 Markdown 渲染。
- 支持代码块高亮。
- 给代码块增加复制按钮。
- 保证普通文本、中文、换行都能正常显示。

完成后效果：

```text
机器人回答代码时，不再是一整坨普通文字，而是有清楚的代码块。
```

重点学习：

- Markdown 是什么。
- 如何安全地把 Markdown 渲染到页面。
- 为什么不能随便把模型返回的 HTML 直接塞进页面。

## 版本 6：提示词配置版

目标：不用改代码，也能切换机器人性格和用途。

要做的功能：

- 把系统提示词从代码里抽出来。
- 可以先放到 `.env` 中，比如 `SYSTEM_PROMPT=你是一个简洁友好的中文助手`。
- 后面可以在页面上做角色选择，比如：

```text
学习助手
编程助手
翻译助手
面试助手
生活助手
```

完成后效果：

```text
同一个聊天框，可以通过不同提示词变成不同用途的机器人。
```

重点学习：

- system prompt 的作用。
- 配置和代码的区别。
- 为什么提示词不一定百分百可靠。

## 版本 7：工具能力版

目标：让机器人不只是聊天，还能拿到真实信息。

可以先做时间工具：

- 后端获取当前时间。
- 把当前时间传给模型。
- 用户问时间时，模型以真实时间回答。

再做天气工具：

- 接入天气 API。
- 用户问天气时，后端先查天气。
- 把天气结果交给模型总结。

后面可以做搜索工具：

- 接入搜索 API。
- 用户问最新新闻、最新资料时，后端先搜索。
- 后端把搜索结果传给模型。
- 模型根据搜索结果回答。

完成后效果：

```text
机器人可以回答实时信息，而不是只靠模型训练时的旧知识。
```

重点学习：

- 大模型为什么默认不知道实时信息。
- 什么是工具调用。
- 后端如何把外部 API 的结果交给模型。

## 版本 8：账号和权限版

目标：让项目更像真实产品。

要做的功能：

- 用户注册和登录。
- 每个用户只能看到自己的聊天记录。
- 后端接口增加权限校验。
- API Key 继续只保存在后端，不能发给前端。

完成后效果：

```text
多人使用时，每个人有自己的聊天记录和设置。
```

重点学习：

- 登录状态是什么。
- Cookie、Session、Token 的区别。
- 为什么前端不能保存服务端 API Key。

## 版本 9：部署上线版

目标：让别人也能访问你的聊天机器人。

要做的功能：

- 整理 README。
- 确认 `.env` 不会上传到 GitHub。
- 选择部署平台。
- 在线上环境配置 `DEEPSEEK_API_KEY`。
- 配置公开访问地址。

完成后效果：

```text
别人可以通过一个网址访问你的聊天机器人。
```

重点学习：

- 本地运行和线上部署的区别。
- 环境变量在服务器上怎么配置。
- 为什么 `.env` 不能上传到 GitHub。

## 推荐迭代顺序

建议不要跳着做，按这个顺序来：

```text
版本 1：基础稳定版
版本 2：多轮对话版
版本 3：停止生成版
版本 4：聊天记录保存版
版本 5：Markdown 和代码显示版
版本 6：提示词配置版
版本 7：工具能力版
版本 8：账号和权限版
版本 9：部署上线版
```

如果只想先做最有价值的三个，优先做：

```text
1. 多轮对话
2. 停止生成
3. 更清楚的错误提示
```

## 每次迭代建议

每次做新功能时，可以按这个流程：

```text
1. 先写清楚要做什么
2. 改后端或前端代码
3. 本地启动项目
4. 浏览器测试
5. 修 bug
6. 更新 README 或学习笔记
7. 提交到 GitHub
```

不要怕项目一开始简单。真实项目也是这样长出来的：先把主流程跑通，再慢慢补状态、错误处理、历史记录、权限、部署。
